			<h1 id="firstHeading" class="firstHeading" lang="en">Symbolic link</h1>
									<div id="bodyContent" class="mw-body-content">
									<div id="siteSub">From Wikipedia, the free encyclopedia</div>
								<div id="contentSub"></div>
												<div id="jump-to-nav" class="mw-jump">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
				</div>
				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><p>In <a href="/wiki/Computing" title="Computing">computing</a>, a <b>symbolic link</b> (also <b>symlink</b> or <b>soft link</b>) is the nickname for any <a href="/wiki/Computer_file" title="Computer file">file</a> that contains a reference to another file or directory in the form of an absolute or relative <a href="/wiki/Path_(computing)" title="Path (computing)">path</a> and that affects pathname resolution.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>
<p>Symbolic links were already present by 1978 in minicomputer operating systems from <a href="/wiki/Digital_Equipment_Corporation" title="Digital Equipment Corporation">DEC</a> and Data General's <a href="/wiki/Data_General_RDOS" title="Data General RDOS">RDOS</a>. Today they are supported by the <a href="/wiki/POSIX" title="POSIX">POSIX</a> operating system standard, most <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> <a href="/wiki/Operating_system" title="Operating system">operating systems</a> such as <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>, <a href="/wiki/Linux" title="Linux">Linux</a>, and <a href="/wiki/Mac_OS_X" class="mw-redirect" title="Mac OS X">Mac OS X</a> and Windows operating systems such as <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a>, <a href="/wiki/Windows_7" title="Windows 7">Windows 7</a> and to some degree in <a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> and <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a> in the form of <a href="/wiki/File_shortcut" title="File shortcut">Shortcut files</a>.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A symbolic link contains a text string that is automatically interpreted and followed by the operating system as a path to another file or directory. This other file or directory is called the "target". The symbolic link is a second file that exists independently of its target. If a symbolic link is deleted, its target remains unaffected. If a symbolic link points to a target, and sometime later that target is moved, renamed or deleted, the symbolic link is not automatically updated or deleted, but continues to exist and still points to the old target, now a non-existing location or file. Symbolic links pointing to moved or non-existing targets are sometimes called <i>broken</i>, <i>orphaned</i>, <i>dead</i>, or <i>dangling</i>.</p>
<p>Symbolic links are different from <a href="/wiki/Hard_link" title="Hard link">hard links</a>. Hard links do not link paths on different <a href="/wiki/Volume_(computing)" title="Volume (computing)">volumes</a> or <a href="/wiki/File_system" title="File system">file systems</a>, whereas symbolic links may point to any file or directory irrespective of the <a href="/wiki/Volume_(computing)" title="Volume (computing)">volumes</a> on which the link and target reside. Hard links always refer to an existing file, whereas symbolic links may contain an arbitrary path that does not point to anything.</p>
<p>Symbolic links operate transparently for many operations: programs that read or write to files named by a symbolic link will behave as if operating directly on the target file. However, they have the effect of changing an otherwise hierarchical filesystem from a <a href="/wiki/Tree_(graph_theory)" title="Tree (graph theory)">tree</a> into a <a href="/wiki/Directed_graph" title="Directed graph">directed graph</a>, which can have consequences for such simple operations as figuring out the current directory of a process. Even the Unix <span style="font-family: monospace, monospace;">..</span> convention for navigating to a directory's parent directory no longer works reliably in the face of symlinks. Some <a href="/wiki/Unix_shell" title="Unix shell">shells</a> <a href="/wiki/Heuristic" title="Heuristic">heuristically</a> try to uphold the illusion of a tree-shaped hierarchy, but when they do, this causes them to produce different results from other programs that manipulate pathnames without such heuristic, relying on the operating system instead.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> Programs that need to handle symbolic links specially (e.g., shells and backup utilities) thus need to identify and manipulate them directly.</p>
<p>Some Unix as well as Linux distributions use symbolic links extensively in an effort to reorder the <a href="/wiki/File_system" title="File system">file system</a> hierarchy. This is accomplished with several mechanisms, such as variant, context-dependent symbolic links. This offers the opportunity to create a more intuitive or application-specific <a href="/wiki/Directory_tree" class="mw-redirect" title="Directory tree">directory tree</a> and to reorganize the system without having to redesign the core set of system functions and utilities.</p>
<h2><span class="mw-headline" id="POSIX_and_Unix-like_operating_systems">POSIX and Unix-like operating systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=2" title="Edit section: POSIX and Unix-like operating systems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In <a href="/wiki/POSIX" title="POSIX">POSIX</a>-compliant operating systems, symbolic links are created with the <code>symlink</code><sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> system call. The <code><a href="/wiki/Ln_(Unix)" title="Ln (Unix)">ln</a></code> shell command normally uses the <code>link</code><sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> system call, which creates a <a href="/wiki/Hard_link" title="Hard link">hard link</a>. When the <code>ln <i>-s</i></code> flag is specified, the symlink() system call is used instead, creating a symbolic link. Symlinks were introduced in <a href="/wiki/Berkeley_Software_Distribution" title="Berkeley Software Distribution">4.2BSD Unix</a> from <a href="/wiki/Computer_Systems_Research_Group" title="Computer Systems Research Group">U.C. Berkeley</a>.</p>
<p>The following command creates a symbolic link at the <a href="/wiki/Command-line_interface" title="Command-line interface">command-line interface</a> (shell):</p>
<pre>
 ln -s target_path link_path
</pre>
<p><tt>target_path</tt> is the relative or absolute path to which the symbolic link should point. Usually the target will exist, although symbolic links may be created to non-existent targets. <tt>link_path</tt> is the path of the symbolic link.</p>
<p>After creating the symbolic link, it may generally be treated as an alias for the target. Any file system management commands (e.g., <code><a href="/wiki/Cp_(Unix)" title="Cp (Unix)">cp</a></code>, <code><a href="/wiki/Rm_(Unix)" title="Rm (Unix)">rm</a></code>) may be used on the symbolic link. Commands which read or write file contents will access the contents of the target file. The <code>rm</code> (delete file) command, however, removes the link itself, not the target file.</p>
<p>The POSIX directory listing application, <code><a href="/wiki/Ls" title="Ls">ls</a></code>, denotes symbolic links with an arrow after the name, pointing to the name of the target file (see following example), when the long directory list is requested (<code>-l</code> option). When a directory listing of a symbolic link that points to a directory is requested, only the link itself will be displayed. In order to obtain a listing of the linked directory, the path must include a trailing directory separator character ('/', slash).</p>
<p>Note: In Below Example do not create "three" directory before creation of link in /tmp directory.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="gp">$</span> mkdir -p /tmp/one/two
<span class="gp">$</span> <span class="nb">echo</span> <span class="s2">"test_a"</span> &gt;/tmp/one/two/a
<span class="gp">$</span> <span class="nb">echo</span> <span class="s2">"test_b"</span> &gt;/tmp/one/two/b
<span class="gp">$</span> <span class="nb">cd</span> /tmp/one/two
<span class="gp">$</span> ls -l
<span class="go">-rw-r--r-- 1 user group 7 Jan 01 10:01 a</span>
<span class="go">-rw-r--r-- 1 user group 7 Jan 01 10:01 b</span>

<span class="gp">$</span> <span class="nb">cd</span> /tmp
<span class="gp">$</span> ln -s /tmp/one/two three
<span class="gp">$</span> ls -l three
<span class="go">lrwxrwxrwx 1 user group 12 Jul 22 10:02 /tmp/three -&gt; /tmp/one/two</span>
<span class="gp">$</span> ls -l three/
<span class="go">-rw-r--r-- 1 user group 7 Jan 01 10:01 a</span>
<span class="go">-rw-r--r-- 1 user group 7 Jan 01 10:01 b</span>

<span class="gp">$</span> <span class="nb">cd</span> three
<span class="gp">$</span> ls -l
<span class="go">-rw-r--r-- 1 user group 7 Jan 01 10:01 a</span>
<span class="go">-rw-r--r-- 1 user group 7 Jan 01 10:01 b</span>
<span class="gp">$</span> cat a
<span class="go">test_a</span>

<h3><span class="mw-headline" id="Storage_of_symbolic_links">Storage of symbolic links</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=3" title="Edit section: Storage of symbolic links">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Early implementations of symbolic links stored the symbolic link information as data in regular files. The file contained the textual reference to the linkâs target, and an indicator<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (April 2010)">clarification needed</span></a></i>]</sup> denoting it as a symbolic link.</p>
<p>This method was slow and an inefficient use of <a href="/wiki/Disk_storage" title="Disk storage">disk-space</a> on small systems. An improvement, called <b>fast symlinks</b>, allowed storage of the target path within the <a href="/wiki/Data_structure" title="Data structure">data structures</a> used for storing file information on disk (<a href="/wiki/Inode" title="Inode">inodes</a>). This space normally stores a list of disk <a href="/wiki/Block_(data_storage)" title="Block (data storage)">block</a> addresses allocated to a file. Thus, symlinks with short target paths are accessed quickly. Systems with fast symlinks often fall back to using the original method if the target path exceeds the available inode space. The original style is <a href="/wiki/Retronym" title="Retronym">retroactively termed</a> a <b>slow symlink</b>. It is also used for disk compatibility with other or older versions of operating systems.</p>
<p>Although storing the link value inside the inode saves a disk block and a disk read, the operating system still needs to parse the path name in the link, which always requires reading additional inodes and generally requires reading other, and potentially many, directories, processing both the list of files and the inodes of each of them until it finds a match with the link's path components. Only when a link points to a file in the same directory do "fast symlinks" provide significantly better performance than other symlinks.</p>
<p>The vast majority of POSIX-compliant implementations use fast symlinks. However, the <a href="/wiki/POSIX" title="POSIX">POSIX</a> standard does not require the entire set of file status information common to regular files to be implemented for symlinks. This allows implementations to use other solutions, such as storing symlink data in directory entries.</p>
<p>The <a href="/wiki/File_system_permissions" title="File system permissions">file system permissions</a> of a symbolic link usually have relevance only to rename or removal operations of the link itself, not to the access modes of the target file which are controlled by the target file's own permissions.</p>
<p>The reported size of a symlink is the number of characters in the path it points to.</p>
<h3><span class="mw-headline" id="Mac_OS_aliases">Mac OS aliases</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=4" title="Edit section: Mac OS aliases">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote">Main article: <a href="/wiki/Alias_(Mac_OS)" title="Alias (Mac OS)">Alias (Mac OS)</a></div>
<p>In Mac OS, applications or users can also employ <i><a href="/wiki/Alias_(Mac_OS)" title="Alias (Mac OS)">aliases</a></i>, which have the added feature of following the target, even if it is moved to another location on the same volume. This is not to be confused with the shell command <a href="/wiki/Alias_(command)" title="Alias (command)">alias</a>.</p>
<h2><span class="mw-headline" id="Microsoft_Windows">Microsoft Windows</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=5" title="Edit section: Microsoft Windows">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="NTFS_symbolic_link">NTFS symbolic link</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=6" title="Edit section: NTFS symbolic link">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote">Main article: <a href="/wiki/NTFS_symbolic_link" title="NTFS symbolic link">NTFS symbolic link</a></div>
<p><a href="/wiki/NTFS" title="NTFS">NTFS</a> 3.1 introduced support symbolic links for any type of file. It was included with <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a>, but was only enabled by default for kernel mode programs; <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> and later versions of Windows enabled support for symbolic links to user mode applications, as well, and supplied the command line utility <code>mklink</code> for creating them. Third-party drivers are required to enable support for NTFS symbolic links in Windows XP.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> Unlike <a href="/wiki/NTFS_junction_point" title="NTFS junction point">junction points</a>, a symbolic link can also point to a file or remote <a href="/wiki/Server_Message_Block" title="Server Message Block">Server Message Block</a> (SMB) network path. Additionally, the NTFS symbolic link implementation provides full support for cross-filesystem links. However, the functionality enabling cross-host symbolic links requires that the remote system also support them.</p>
<p>Symbolic links are designed to aid in migration and application compatibility with <a href="/wiki/POSIX" title="POSIX">POSIX</a> operating systems. Microsoft aimed for Windows Vista's symbolic links to "function just like UNIX links".<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> However, the implementation varies from Unix symbolic links in several ways. For example, Windows Vista users must manually indicate when creating a symbolic link whether it is a file or a directory.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> Windows 7 and Vista support a maximum of 31 <a href="/wiki/Reparse_point" class="mw-redirect" title="Reparse point">reparse points</a> (and therefore symbolic links) for a given path (i.e. any given path can have at most 31 indirections before Windows gives up).<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> Only users with the new <i>Create Symbolic Link</i> privilege, which only administrators have by default, can create symbolic links.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> If this is not the desired behavior, it must be changed in the Local Security Policy management console. Additionally, NTFS symbolic links to files are distinct from NTFS symbolic links to directories and therefore cannot be used interchangeably, unlike on POSIX where the same symbolic link can refer to either files or directories.</p>
<p>In Windows Vista and later, when the working directory path ends with a symbolic link, the current parent path reference, <code class="mw-highlight" id="" style="" dir="ltr">..</code>, will refer to the parent directory of the symbolic link rather than that of its target. This behaviour is also found at the shell level in at least some POSIX systems, including <a href="/wiki/Linux" title="Linux">Linux</a>, but never in accessing files and directories through operating system calls. For instance, bash builtin commands <code class="mw-highlight" id="" style="" dir="ltr">pwd</code> and <code class="mw-highlight" id="" style="" dir="ltr">cd</code> operate on the current logical directory. <code class="mw-highlight" id="" style="" dir="ltr">pwd</code> is often used in scripts to determine the actual current working directory. When any path is used with a system call, any use of <code class="mw-highlight" id="" style="" dir="ltr">..</code> will use the actual filesystem parent of the directory containing the <code class="mw-highlight" id="" style="" dir="ltr">..</code> pseudo-directory entry. So, <code class="mw-highlight" id="" style="" dir="ltr">cd ..; cat something</code> and <code class="mw-highlight" id="" style="" dir="ltr">cat ../something</code> may return completely different results.</p>
<p>Example:</p>
<pre>
mklink /D "C:\Users\&lt;yourusername&gt;\Appdata\Local\Plex Media Server" "G:\Plex Media Server"
</pre>
<h3><span class="mw-headline" id="NTFS_junction_points">NTFS junction points</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=7" title="Edit section: NTFS junction points">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote">Main article: <a href="/wiki/NTFS_junction_point" title="NTFS junction point">NTFS junction point</a></div>
<p>The <a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> version of <a href="/wiki/NTFS" title="NTFS">NTFS</a> introduced <a href="/wiki/NTFS_reparse_point" title="NTFS reparse point">reparse points</a>, which enabled, among other things, the use of <a href="/wiki/Volume_Mount_Point" class="mw-redirect" title="Volume Mount Point">Volume Mount Points</a> and junction points. Junction points are for directories only, and moreover, local directories only; junction points to remote shares are unsupported.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup> The Windows 2000 and XP Resource Kits include a program called linkd to create junction points; a more powerful one named Junction was distributed by <a href="/wiki/Sysinternals" title="Sysinternals">Sysinternals</a>' <a href="/wiki/Mark_Russinovich" title="Mark Russinovich">Mark Russinovich</a>.</p>
<p>Not all standard applications support reparse points. Most noticeably, Backup suffers from this problem and will issue an error message 0x80070003<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> when the folders to be backed up contain a reparse point.</p>
<h3><span class="mw-headline" id="Shortcuts">Shortcuts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=8" title="Edit section: Shortcuts">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Computer_shortcut" class="mw-redirect" title="Computer shortcut">Shortcuts</a>, which are supported by the graphical file browsers of some operating systems, may resemble symbolic links but differ in a number of important ways. One difference is what type of software is able to follow them:</p>
<ul>
<li>Symbolic links are automatically resolved by the file system. Any software program, upon accessing a symbolic link, will see the target instead, whether the program is aware of symbolic links or not.</li>
<li>Shortcuts are treated like ordinary files by the file system and by software programs that are not aware of them. Only software programs that understand shortcuts (such as the Windows shell and file browsers) treat them as references to other files.</li>
</ul>
<p>Another difference are the capabilities of the mechanism:</p>
<ul>
<li><a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> shortcuts normally refer to a destination by an <a href="/wiki/Absolute_path" class="mw-redirect" title="Absolute path">absolute path</a> (starting from the <a href="/wiki/Root_directory" title="Root directory">root directory</a>), whereas POSIX symbolic links can refer to destinations via either an absolute or a <a href="/wiki/Relative_path" class="mw-redirect" title="Relative path">relative path</a>. The latter is useful if both the location and destination of the symbolic link share a common path <a href="/wiki/Prefix" title="Prefix">prefix</a>, but that prefix is not yet known when the symbolic link is created (e.g., in an <a href="/wiki/Archive_file" title="Archive file">archive file</a> that can be unpacked anywhere).</li>
<li>Microsoft Windows application shortcuts contain additional metadata that can be associated with the destination, whereas POSIX symbolic links are just strings that will be interpreted as absolute or relative pathnames.</li>
<li>Unlike symbolic links, Windows shortcuts maintain their references to their targets even when the target is moved or renamed. Windows domain clients may subscribe to a <a href="/wiki/Windows_service" title="Windows service">Windows service</a> called <i><a rel="nofollow" class="external text" href="http://technet.microsoft.com/en-us/library/cc778822%28v=ws.10%29.aspx">Distributed Link Tracking</a></i> to track the changes in files and folders to which they are interested. The service maintains the integrity of shortcuts, even when files and folders are moved across the network.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup> Additionally, in Windows 9x and later, <a href="/wiki/Windows_shell" title="Windows shell">Windows shell</a> tries to find the target of a broken shortcut before proposing to delete it.</li>
</ul>
<h3><span class="mw-headline" id="Folder_shortcuts">Folder shortcuts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=9" title="Edit section: Folder shortcuts">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Almost like shortcuts, but transparent to the Windows shell.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup> They are implemented as ordinary folders (which need to have the <i>read only</i> and/or <i>system</i> attribute<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">[14]</a></sup>) containing a shortcut named <i>target.lnk</i> which refers to the target and a (hidden) <i>desktop.ini</i> with (at least) the following contents:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="k">[.ShellClassInfo]</span>
 <span class="na">CLSID2</span><span class="o">=</span><span class="s">{0AFACED1-E828-11D1-9187-B532F1E9575D}</span>
</pre></div>
<p>Folder shortcuts are created and used from the Windows shell in the <i>network neighborhood</i> for example.</p>
<h3><span class="mw-headline" id="Shell_objects">Shell objects</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=10" title="Edit section: Shell objects">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>shell objects</i><sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup> or <i>shell folders</i> are defined in the Windows registry and can be used to implement a sort of symbolic link too. Like folder shortcuts, they are transparent to the Windows shell.</p>
<p>A minimal implementation is (the CLSID <i>{00000000-0000-0000-0000-000000000000}</i> is used as a placeholder):</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\CLSID\{00000000-0000-0000-0000-000000000000}]</span>
 <span class="na">@</span><span class="o">=</span><span class="s">"display name"</span>
 <span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\CLSID\{00000000-0000-0000-0000-000000000000}\DefaultIcon]</span>
 <span class="na">@</span><span class="o">=</span><span class="s">"..." ; path to icon</span>
 <span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\CLSID\{00000000-0000-0000-0000-000000000000}\InProcServer32]</span>
 <span class="na">@</span><span class="o">=</span><span class="s">"%SystemRoot%\\System32\\ShDocVw.Dll"</span>
 <span class="na">"ThreadingModel"</span><span class="o">=</span><span class="s">"Apartment"</span>
 <span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\CLSID\{00000000-0000-0000-0000-000000000000}\Instance]</span>
 <span class="na">"CLSID"</span><span class="o">=</span><span class="s">"{0AFACED1-E828-11D1-9187-B532F1E9575D}"</span>
 <span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\CLSID\{00000000-0000-0000-0000-000000000000}\Instance\InitPropertyBag]</span>
 <span class="na">"Attributes"</span><span class="o">=</span><span class="nv">hex</span><span class="p">:</span><span class="m">15,00,00,00</span>
 <span class="na">"Target"</span><span class="o">=</span><span class="s">"..." ; absolute (WITHOUT "TargetKnownFolder" or "TargetSpecialFolder" only)</span>
                <span class="c1">; or relative path to target</span>
 <span class="na">"TargetKnownFolder"</span><span class="o">=</span><span class="s">"{guidguid-guid-guid-guid-guidguidguid}" ; GUID of target folder, Windows Vista and later</span>
 <span class="na">"TargetSpecialFolder"</span><span class="o">=</span><span class="s">"0x00xy" ; CSIDL of target</span>
 <span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\CLSID\{00000000-0000-0000-0000-000000000000}\ShellFolder]</span>
 <span class="na">"Attributes"</span><span class="o">=</span><span class="nv">hex</span><span class="p">:</span><span class="m">00,00,00,00</span>
</pre></div>
<p>The <i>My Documents</i> folder on the <i>Desktop</i> as well as the <i>Fonts</i> and the <i>Administrative Tools</i> folders in the <i>Control Panel</i> are examples of <i>shell objects</i> redirected to file-system folders.</p>
<h3><span class="mw-headline" id="Cygwin_symbolic_links">Cygwin symbolic links</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=11" title="Edit section: Cygwin symbolic links">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Cygwin" title="Cygwin">Cygwin</a> simulates POSIX-compliant symbolic links in the Microsoft Windows file system. It uses identical programming and user utility interfaces as Unix (see above), but creates Windows shortcuts (.lnk files) with additional information used by Cygwin at the time of symlink resolution. Cygwin symlinks are compliant with both Windows and the POSIX standard.<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (April 2010)">clarification needed</span></a></i>]</sup></p>
<p>Additionally, Cygwin can be set up to support native Windows symbolic links which can be used out of Cygwin without restrictions.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup> This requires:</p>
<ol>
<li>Editing CYGWIN variable (or use another way to set Cygwin option);</li>
<li>Running the Cygwin with elevated rights because Windows restricts the creation of symbolic links to privileged users</li>
</ol>
<p>Some differences exist, however. Cygwin has no way to specify shortcut-related information â such as working directory or icon â as there is no place for such parameters in <code>ln -s</code> command. To create standard Microsoft .lnk files Cygwin provides the <code>mkshortcut</code> and <code>readshortcut</code> utilities<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">[17]</a></sup></p>
<p>The Cygwin User's Guide<sup id="cite_ref-18" class="reference"><a href="#cite_note-18">[18]</a></sup> has more information on this topic.</p>

<h2><span class="mw-headline" id="Amiga">Amiga</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=13" title="Edit section: Amiga">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The command creating symbolic links is <code>makelink</code>, which is also used for hard links. Internally the dos.library returns an error code indicating that a target is a soft link if you try to perform actions on it that are only legal for a file, and applications that wish to follow the symbolic link then needs to explicitly make a call to follow the link and retry the operation. The <a href="/wiki/AmigaDOS" title="AmigaDOS">AmigaDOS</a> shell will follow links automatically.</p>
<h2><span class="mw-headline" id="OS.2F2">OS/2</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=14" title="Edit section: OS/2">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the <a href="/wiki/OS/2" title="OS/2">OS/2</a> operating system, symbolic links somewhat resemble <a href="/wiki/Shadow_(OS/2)" title="Shadow (OS/2)">shadows</a> in the graphical <a href="/wiki/Workplace_Shell" title="Workplace Shell">Workplace Shell</a>. However, shadows, due to the fully object-oriented System Object Model, are considerably more powerful and robust than a simple link. For example, shadows do not lose their capabilities when renamed or when either the object or subject of the link is relocated.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (April 2012)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="Variable_symbolic_links">Variable symbolic links</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Symbolic_link&amp;action=edit&amp;section=15" title="Edit section: Variable symbolic links">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Symbolic links may be implemented in a context-dependent or variable fashion, such that the link points to varying targets depending on a configuration parameter, run-time parameter, or other instantaneous condition.</p>
<p>A <i>variable</i> or <i>variant symbolic link</i> is a symbolic link that has a variable name embedded in it. This allows some flexibility in filesystem order that is not possible with a standard symbolic link. Variables embedded in a symbolic link may include user and environment specific information.</p>
<p><a href="/wiki/Operating_system" title="Operating system">Operating systems</a> that make use of variant symbolic links include <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>, <a href="/wiki/NetBSD" title="NetBSD">NetBSD</a>, <a href="/wiki/DragonFly_BSD" title="DragonFly BSD">DragonFly BSD</a>, <a href="/wiki/Domain/OS" title="Domain/OS">Domain/OS</a>.<sup id="cite_ref-21" class="reference"><a href="#cite_note-21">[21]</a></sup></p>
<p><a href="/wiki/Tru64" class="mw-redirect" title="Tru64">Tru64</a> uses a <i>context dependent symbolic link</i> where the context is the cluster member number.</p>
<p><a href="/wiki/Pyramid_Technology" title="Pyramid Technology">Pyramid Technology</a>'s OSx <a href="/wiki/Operating_System" class="mw-redirect" title="Operating System">Operating System</a> implemented <i>conditional symbolic links</i> which pointed to different locations depending on which <a href="/wiki/Universe_(Unix)" title="Universe (Unix)">universe</a> a program was running in. The universes supported were <a href="/wiki/AT%26T_Corporation" title="AT&amp;T Corporation">AT&amp;T</a>'s <a href="/wiki/UNIX_System_V" title="UNIX System V">SysV.3</a> and the <a href="/wiki/Berkeley_Software_Distribution" title="Berkeley Software Distribution">Berkeley Software Distribution</a> (BSD 4.3). For example: if the <a href="/wiki/Ps_(Unix)" title="Ps (Unix)">ps</a> command was run in the <i>att</i> universe, then the symbolic link for the directory <i>/bin</i> would point to <i>/.attbin</i> and the program <i>/.attbin/ps</i> would be executed. Whereas if the ps command was run in the <i>ucb</i> universe, then <i>/bin</i> would point to <i>/.ucbbin</i> and <i>/.ucbbin/ps</i> would be executed. Similar Conditional Symbolic Links were also created for other directories such as <i>/lib</i>, <i>/usr/lib</i>, <i>/usr/include</i>.</p>
